# AI / 开发协作指南

## 项目目标

MergeWong 是 Go + Svelte 的中小型数据库同步服务。实现优先级依次是：正确性、可恢复性、可观测性、性能、连接器数量。不要把“定时轮询增量”描述成 CDC 实时同步。

## 修改前先读

- `README.md`：产品范围和当前能力；
- `doc/ARCHITECTURE.md`：现状、风险和目标架构；
- `doc/ROADMAP.md`：迭代顺序。

## 代码边界

- `internal/services/sync_service.go` 是当前原型同步核心，重构时保持 handler API 可迁移。
- 元数据 CRUD 可以使用 GORM；数据同步链路优先使用 `database/sql` 和显式方言适配。
- 所有同步进度必须写入独立检查点，不能依赖目标表 `MAX()`。
- 目标写入按 at-least-once 设计，必须通过主键 upsert 保证幂等。
- 表名和列名不能作为值参数绑定，必须经过白名单校验和方言引用。
- 新增数据库类型时，Reader、Writer 和元数据库支持要分别声明，不要笼统声称“已支持”。
- 不要提交真实数据库密码、JWT secret、生成日志、构建产物或本地配置。

## 同步语义检查清单

涉及同步代码的变更必须回答：

1. 检查点在何时推进？
2. 批次成功但检查点写入前崩溃会怎样？
3. 相同增量值的多行如何稳定分页？
4. 重跑是否幂等？
5. 更新和删除如何表达？
6. 同一任务能否重叠执行？
7. 大表是否存在全表载入或长事务？

## 验证

```bash
go test ./...
cd web && npm ci && npm run build
```

同步语义变更还应增加数据库集成测试，至少覆盖：初始全量、增量新增、已有行更新、软/物理删除策略、同游标值多行、批次中断恢复和重复执行。

## 已知遗留

项目从 `apiwong` 复制而来，Go module、import、二进制名、Docker 名称、页面标题及部分文档仍有旧名称。统一重命名应作为一个独立变更完成，并在改后运行全量编译和容器构建验证。
